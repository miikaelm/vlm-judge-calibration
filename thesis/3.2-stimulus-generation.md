# 3.2 Stimulus Generation

Each stimulus in this study is a triple of images: a **source** image (the original, unedited document), a **ground truth** image (the source with the intended edit applied correctly), and a **degraded** image (the source with the intended edit applied but with a controlled error introduced). Together, these three images record a single editing scenario at a known, parameterised deviation from correctness. The stimulus generation pipeline has five stages: base layout instantiation, HTML-layer degradation, browser rendering, image-layer degradation (for a subset of configs), and metadata serialisation.

## 3.2.1 Rendering Pipeline

All images are rendered from self-contained HTML/CSS documents using Playwright, a headless Chromium-based browser automation library. HTML/CSS is the generation medium because it offers direct programmatic control over all the visual properties we manipulate — element position, font size, colour, weight, style, letter spacing, and text content — without the opacity of generative models. A CSS override block injected into the document's `<head>` is sufficient to apply any of the quantitative degradations described below, and the browser's layout engine handles all rendering geometry consistently.

The rendering configuration is kept fixed throughout the study:
- **Viewport:** 1024 × 1024 pixels, device scale factor 1.0 (no HiDPI scaling).
- **Font rendering:** Subpixel rendering and LCD text are disabled (`--disable-lcd-text`, `--font-render-hinting=none`) to eliminate platform-dependent anti-aliasing differences that could introduce uncontrolled variation.
- **Animations:** Disabled globally via CSS (`animation-duration: 0s`, `transition-duration: 0s`) so the rendered state is always the fully settled layout.
- **Network:** All fonts and assets are loaded from within the document or via a fixed URL seed (for the one image-background layout); `wait_until="networkidle"` and `document.fonts.ready` are awaited before screenshot capture to ensure all assets resolve.
- **Output format:** PNG, then downscaled to 512 × 512 using Lanczos resampling for storage and API submission.

This approach produces pixel-perfect, deterministic renders: given the same HTML string and the same Chromium version, the output is identical on any machine. There are no generative stochastic components in the image creation process.

## 3.2.2 Layouts and Difficulty Tiers

Rather than free-form HTML templates, the stimulus set is organised around a fixed registry of **layouts**, self-contained scene definitions that specify a set of named text roles, their default typographic styles, the HTML structure, and which edit types each layout supports. Layouts are the unit of visual variety in the stimulus set.

Eleven layouts are defined, spanning two difficulty tiers:

| Tier | Description | Layouts |
|------|-------------|---------|
| **Easy** | Large text (≥48 px), high contrast, clean solid or simple gradient background | `solo_headline`, `gradient_headline`, `pattern_headline` |
| **Medium** | Moderate text (26–60 px), more complex backgrounds, multi-element scenes, absolutely positioned elements | `title_byline`, `name_card`, `quote_attribution`, `corner_badge`, `banner_caption`, `image_bg_headline`, `title_subtitle`, `header_body` |

The difficulty tiers are intentional design choices, not emergent categories. Easy-tier layouts are designed so that every degradation should in principle be detectable: the text is large, the background is simple, and there is nothing to confuse the element of interest. Medium-tier layouts introduce multiple text elements and more complex backgrounds, which may interact with VLM sensitivity independently of the degradation magnitude.

Each layout is parameterised by a `LAYOUT_SETS` mapping that constrains which layouts are used for each `(edit_type, difficulty)` cell in the stimulus design matrix. This prevents applying an edit to a layout that does not support it (e.g., a rotation edit on a layout whose constraints forbid rotation because adjacent elements would be occluded). Layout-level position-offset constraints further restrict which displacement directions are allowed per element to avoid text reflow, a CSS `transform: translate()` is used for all spatial offsets precisely because it does not trigger document reflow and therefore does not disturb any other layout element.

## 3.2.3 Edit Types

Stimuli are generated by crossing a set of edit types with a set of degradation configs. The edit types represent the operations that a text-editing model would be asked to perform:

- **Color change:** Change the text colour of a named element to a target hex value.
- **Scale change:** Resize the text of a named element by a relative amount (e.g., 20% larger, 30% smaller).
- **Rotation:** Rotate a named element by a target angle in degrees.
- **Relocation / alignment:** Move a named element to a different position on the canvas (absolute position or alignment grid position).
- **Font weight:** Change the CSS `font-weight` property of a named element.
- **Font style (italic):** Change the CSS `font-style` property of a named element.
- **Letter spacing:** Change the CSS `letter-spacing` property of a named element.
- **Font family:** Substitute the typeface of a named element.

For each stimulus, the "correct" edit is applied first (producing the ground truth image) and then the degradation is applied on top of the correctly edited HTML (producing the degraded image). The source image is rendered from the unmodified base layout. This three-image structure ensures that the degraded image reflects both the intended edit and the error, replicating the situation in which an editing model almost-but-not-quite executes an instruction.

## 3.2.4 Degradation Dimensions and Magnitude Parameterisation

The degradation engine applies controlled errors to the HTML/CSS source before rendering. All HTML-layer degradations operate by injecting a CSS override block (`<style>#element_id { property: value !important; }</style>`) immediately before the `</head>` tag. The `!important` qualifier ensures the override takes precedence over the layout's base styles. Twelve degradation dimensions are implemented at the HTML layer, plus three at the image layer (applied to the rendered PNG after browser output).

### HTML-Layer Degradations

**Color offset.** The element's CSS `color` property is overridden with a colour that deviates from the correct colour by a specified ΔE (CIEDE2000) distance. The shift is applied along the CIE L\* (lightness) axis, which approximates a uniform perceptual deviation. The target ΔE is first applied as a positive lightness shift; if that clips to the gamut boundary (L\* > 100 or < 0), the shift is reversed. Ten magnitude tiers are defined, spanning ΔE ≈ 0.5–50:

| Magnitude label | ΔE range (CIEDE2000) | Perceptual interpretation |
|-----------------|----------------------|--------------------------|
| micro | 0.5 – 1.5 | Sub-threshold; not perceptible side-by-side |
| tiny | 1.5 – 2.5 | Near-threshold; barely perceptible |
| subtle | 2.5 – 4.0 | Subtle; perceptible on close comparison |
| small | 4.0 – 6.5 | Just-noticeable difference (JND range) |
| moderate | 6.5 – 9.0 | Clearly visible on comparison |
| medium | 9.0 – 13.0 | Clearly wrong colour |
| strong | 13.0 – 18.0 | Strong deviation |
| large | 18.0 – 26.0 | Large deviation |
| very\_large | 26.0 – 36.0 | Very large deviation |
| xlarge | 36.0 – 50.0 | Obviously wrong colour |

**Position offset.** The element's visual position is shifted using a CSS `transform: translate(Xpx, Ypx)` override. Twelve configs cover four directions (right, left, down, up) at three magnitude tiers (small: 3–10 px; medium: 11–50 px; large: 51–150 px). Using `transform: translate` rather than `position` or `margin` modifications ensures the offset is always relative to the element's rendered position, regardless of the layout's flexbox or absolute positioning, and does not trigger reflow.

**Scale error.** The element's `font-size` is overridden at ±3% to ±90% of the correct value. Ten configs cover upward and downward errors at five tiers each (small: 2–5%; medium: 6–14%; large: 15–35%; xlarge: 36–65%; xxlarge: 66–90%). The base font size is extracted from the HTML by regex parsing of the element's existing style block, so the error percentage is applied relative to the actual correct size, not a fixed baseline.

**Rotation.** The element is rotated by applying `transform: rotate(Ndeg)`. Ten configs cover clockwise and counter-clockwise rotations at five tiers (tiny: ~1°; small: ~3°; medium: ~5°; large: ~15°; xlarge: ~22°).

**Font weight.** The `font-weight` property is overridden to one of nine CSS numeric values (100–900 in steps of 100). No jitter is applied to categorical properties; the magnitude is the exact weight value.

**Font style.** The `font-style` property is set to `italic` or `oblique`. Two configs.

**Letter spacing.** The `letter-spacing` property is overridden in the range −3 to +15 px. Five configs cover positive expansion (small: 1–3 px; medium: 3–8 px; large: 8–15 px) and negative compression (small: −1.5 to −0.5 px; medium: −3 to −1.5 px).

**Opacity.** The `opacity` property is reduced from 1.0 to approximately 0.30. Zero opacity (invisible text) is excluded; the minimum is ~0.30 (~70% transparent).

**Font family.** Seven alternative typefaces replace the layout's base font: three sans-serif (Arial, Verdana, Trebuchet MS), two serif (Georgia, Palatino), one monospace (Courier New), and one condensed display (Impact). All are system fonts available in Chromium's rendering environment.

### Image-Layer Degradations

Three degradation types are applied as pixel-level post-processing to the rendered PNG, using Pillow (PIL):

- **Gaussian noise:** Additive noise with standard deviation σ ∈ {3–8, 9–22, 23–40} (small, medium, large).
- **JPEG compression:** The PNG is re-saved as JPEG at quality ∈ {72–90, 40–65, 10–35} and then reloaded, introducing block artefacts (light, medium, heavy).
- **Gaussian blur:** A Gaussian filter with radius ∈ {0.5–2.0, 2.1–4.0, 4.1–7.0} (small, medium, large).

These three dimensions serve as a reference baseline: they are not plausible errors for a text-editing model operating on HTML (which produces clean rasterised output), but they represent image quality degradations that VLMs encounter in natural images and that human annotators would easily detect at moderate magnitudes.

## 3.2.5 Continuous Jitter Sampling

To avoid over-fitting the sensitivity curve to a fixed set of exact magnitude values, each numeric degradation config specifies a **jitter range** (`min_X` / `max_X`) rather than a point value. At generation time, the actual parameter value is sampled uniformly from the range and stored in the stimulus metadata. This means that stimuli within the same magnitude tier differ in their exact values, and the sensitivity curve is fit over these continuous observations rather than over binned means. The magnitude tier label (e.g., "small", "medium") is preserved in the metadata as a grouping key for display; the resolved numeric value is used in all statistical analyses.

The tier boundaries were chosen so that:
- The smallest tier (e.g., ΔE 0.5–1.5 for color) falls below the expected human just-noticeable difference, providing a floor for the sensitivity curve.
- The top tier (e.g., ΔE 36–50) produces errors that are unambiguously visible even without a reference image.
- Consecutive tier ranges are non-overlapping, so the ordering of stimuli along the magnitude axis is unambiguous.

The tier labels "micro", "tiny", "subtle", and "small" correspond to a rough perceptual interpretation: micro is sub-threshold even on direct side-by-side comparison; small corresponds to the just-noticeable-difference range; tiers above small (moderate through xlarge) are progressively more visible, with xlarge corresponding to errors apparent even when only the degraded image is viewed in isolation.

## 3.2.6 Seed Management

The stimulus generation script uses a per-instance seeded random number generator, constructed from the stimulus ID using Python's `random.Random(stimulus_id)`. This ensures that jitter values are deterministic given the stimulus ID, so the stimulus set is fully reproducible: re-running generation with the same configs and templates produces identical parameter values. The seed is derived from the stimulus ID rather than a global seed to avoid subprocess RNG inheritance issues when generation is parallelised.

## 3.2.7 Sample Size and Stimulus Coverage

The generation process produced 703 stimuli recorded in `data/generated/manifest.jsonl`. This represents a practical sample from the full factorial of (edit types) × (degradation configs) × (layouts), which would be too large to evaluate against multiple VLMs within the study's API budget constraints.

The stimulus set is not balanced: not every (dimension, edit_type, layout) cell is equally populated. The design prioritises depth over breadth within dimensions: each degradation dimension is covered across multiple magnitude tiers and at least two or three layout variants (to distinguish magnitude effects from layout-specific confounds), but not all combinations of edit type and degradation are represented.

For the VLM evaluation, two analysis approaches are available depending on the research question:

1. **Model-specific corpora:** Use all stimuli for which a given model returned a valid (parseable) response. The per-model corpus may differ in size and composition if some stimuli exceeded token limits or encountered API failures for a subset of models. This approach maximises statistical power for each model separately, at the cost of strict comparability across models.

2. **Common corpus:** Restrict analysis to the subset of stimuli for which all evaluated models returned valid responses. This approach enables direct cross-model comparison on identical stimuli, at the cost of reduced corpus size.

The choice between approaches is discussed in Section 3.4 in the context of specific analyses.

At the current sample size (~700 stimuli across ~10 magnitude tiers per dimension and ~8 dimensions), per-cell counts range from approximately 5 to 30 stimuli depending on the dimension and difficulty tier. This is sufficient to estimate mean scores and detect strong monotone trends in the sensitivity curve, but may be underpowered for detecting subtle slope differences between models or edit types within a single dimension.
