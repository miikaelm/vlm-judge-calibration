# 3.3 Evaluation Protocol

## 3.3.1 Models Under Evaluation

The following proprietary VLMs were evaluated as judges in this study. All models were accessed via their respective cloud APIs. Inference was deterministic where supported: temperature was set to 0.0 for all models to minimise response variance unrelated to the stimulus.

| Model | API | Temperature | Max tokens |
|-------|-----|-------------|------------|
| GPT-4o | OpenAI Chat Completions API | 0.0 | 1024 |
| Gemini 3.1 Pro Preview | Google Generative AI Batch API | 0.0 | 1024 |
| Qwen2.5-VL | Local | 0.0 | 1024 |

**Image encoding.** For the OpenAI API, images are encoded as base64 data URIs with `detail: "high"`, which instructs the model to use the full tile-based resolution processing (up to 2048 px longest side, 512 px tiles). For the Gemini API, images are passed as inline base64-encoded data in the batch JSONL format.

**Cost and access-tier constraints.** Gemini models were accessed via the Batch API at 50% of the standard per-token cost. This was a practical necessity: Tier 1 API accounts are subject to a daily request cap (250 requests per day for some Gemini Pro variants), which would have made synchronous evaluation of the full stimulus set across two experiments prohibitively slow. The Batch API bypasses this limit at the cost of asynchronous execution (jobs typically completing within 24 hours). GPT-4o was accessed via a Tier 1 OpenAI account with standard rate limits; the synchronous runner handles rate limiting through sequential submission with implicit per-request delays.

## 3.3.2 Evaluation Prompt Design

The prompt structure is aligned with the evaluation protocols used in TextEditBench to ensure that our calibration findings are directly interpretable in terms of those benchmarks' judge setup.

All prompts are stored in `configs/vlm_prompts.yaml` and loaded at runtime. The response format is structured JSON throughout: both the system message and the user template instruct the model to return **only** a valid JSON object with no prose, no markdown fences, and no explanation. This design was chosen over free-text responses to enable robust automatic parsing and to avoid variability in score extraction.

### Experiment 1: Perceptual Sensitivity Prompt

**Images submitted:** (1) the ground truth image, (2) the degraded image.  
**No edit instruction is provided.** The model is asked only to compare the two images visually.

System message:
> You are an expert visual editor and typographic evaluator. Your task is to compare two document images and determine whether they differ visually. Respond ONLY with valid JSON — no prose, no markdown fences, no explanation.

User template:
> I will show you two images in order:
> 1. A REFERENCE image (the correct, intended result)
> 2. A COMPARISON image (a version that may or may not contain subtle errors)
>
> Determine whether there is any, even slight, difference between the two images.
>
> Return ONLY this exact JSON object:
> `{"detected_difference": <true or false>, "similarity_score": <integer 1-5, where 1=very different and 5=identical>, "description": "<brief description of differences, or empty string if none>"}`

This prompt elicits three outputs per stimulus: a binary detection flag (`detected_difference`), a graded similarity score (`similarity_score`, 1–5), and a free-text description of any observed differences.

### Experiment 2: Edit Judgment Prompt

**Images submitted:** (1) the source (original, unedited) image, (2) the degraded image.  
**The edit instruction is interpolated into the user template.** The model is asked to evaluate the edit result against the instruction, without being shown the ground truth.

System message:
> You are an expert visual editor and typographic evaluator. Your task is to evaluate the quality of a text editing operation applied to a document image. You will be shown an original source image, an edit instruction, and a result image. Evaluate the result strictly based on how accurately the edit was performed. Respond ONLY with valid JSON — no prose, no markdown fences, no explanation.

User template:
> Edit instruction: {instruction}
>
> I will show you two images in order:
> 1. The ORIGINAL image (before any edit)
> 2. The RESULT image (the edit was applied — may be correct, slightly off, or clearly wrong)
>
> Rate the result on each dimension using a 1–5 integer scale: [five dimensions listed with anchors]
>
> Return ONLY this exact JSON object:
> `{"instruction_following": <1-5>, "text_accuracy": <1-5>, "visual_consistency": <1-5>, "layout_preservation": <1-5>, "overall_quality": <1-5>, "errors_noticed": "<brief description>"}`

The five scoring dimensions are drawn from TextEditBench's semantic evaluation rubric:

| Dimension | What it measures | Score anchors |
|-----------|-----------------|---------------|
| `instruction_following` | Did the edit match what was requested, with no unintended changes? | 1 = wrong operation or not executed; 5 = exactly performed |
| `text_accuracy` | Is the text content correct in wording, spelling, and case? | 1 = wrong or missing; 5 = exact and correct |
| `visual_consistency` | Were all visual properties not part of the requested change preserved? | 1 = widespread collateral damage; 5 = only requested property changed |
| `layout_preservation` | Did non-target areas of the image remain unchanged? | 1 = large portions changed; 5 = non-target areas fully preserved |
| `overall_quality` | Holistic quality of the editing result | 1 = unusable; 5 = excellent |

The `errors_noticed` field is not scored; it is used as qualitative data and for post-hoc validation that the model's scores reflect the degradation that was introduced.

### Separated Rubric Variant (Reference)

A separated variant of the Experiment 2 prompt (`experiment_2_separated`) was implemented for comparison with TextEditBench's evaluation approach. In this variant, each rubric dimension is queried in a separate API call with a distinct system prompt focused exclusively on that dimension, returning `{"score": <1-5>, "rationale": "<explanation>"}`. This produces five API calls per stimulus instead of one. The separated variant was implemented to enable comparison with benchmarks that use dimension-by-dimension querying, but the primary analysis in this study uses the combined prompt to match the more common single-call protocol.

## 3.3.3 Response Parsing

VLM responses are parsed into structured data by a three-strategy parser:

1. **Direct parse:** Attempt `json.loads()` on the stripped response string. This succeeds for well-behaved responses.
2. **Fence extraction:** If direct parsing fails, locate a ```` ```json ... ``` ```` or ```` ``` ... ``` ```` code block using a regex and attempt to parse its contents. This recovers responses where the model wrapped its JSON in markdown formatting despite the instruction not to.
3. **Brace extraction:** If fence extraction also fails, locate the first `{...}` block in the response using a regex and attempt to parse it. This recovers responses with leading or trailing prose.

If all three strategies fail, the response is recorded as a parse failure and logged to `data/parse_failures.jsonl`. Parse failures are excluded from all analyses. Numeric scores extracted from the JSON are validated: only integer values in [1, 5] are accepted; values outside this range are treated as missing for that dimension.

## 3.3.4 Experimental Conditions

Each stimulus is evaluated in both experiments (Exp1 and Exp2) for each model. The evaluation runner appends results to a JSONL file immediately after each API call (crash-safe streaming output), allowing interrupted runs to resume without duplicating already-completed calls. Completed stimulus IDs are tracked and excluded from subsequent runs via the `--filter-ids` mechanism, which reads a completed results file and restricts the new run to stimuli not yet present.

## 3.3.5 Noop and Perfect Control Conditions

Two control conditions test for false positives, cases where the VLM reports errors that are not present.

The **noop condition** uses an identical image pair: the degraded image is byte-identical to the ground truth (no error introduced). It applies to both experiments.

- In **Experiment 1**, the VLM is shown two identical images and should report `detected_difference: false` and `similarity_score: 5`. Any reported difference is a false positive.
- In **Experiment 2**, the VLM is shown the source image alongside an unchanged copy as the "edited" result. The expected score is 0 as no edit was applied. High scores indicate the VLM failed to detect that no edit was performed.

The **perfect condition** applies only to Experiment 2: the VLM is shown the source image and a correctly executed edit (ground truth). High scores across all dimensions are expected; low scores indicate the VLM is penalising correctly executed edits.

Control results are stored separately from degraded results and analysed independently (Section 3.4.4).
